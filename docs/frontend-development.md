# Frontend Development Guide

## Overview

This project uses **Mock Service Worker (MSW)** to enable frontend development without a running backend. MSW intercepts network requests at the Service Worker level in the browser, returning realistic mock data. This means:

- Frontend teams can develop independently of backend teams
- Loading states, errors, and caching behave exactly as in production
- The same mock handlers can be reused for automated tests
- Switching to the real backend requires zero code changes

## OptamusUI — Mock Mode

### How it works

MSW is automatically enabled when running `npm run dev`. The initialization happens in `src/routes/+layout.svelte`:

```ts
if (dev) {
  const { worker } = await import('$lib/mocks');
  await worker.start({ onUnhandledRequest: 'bypass' });
}
```

- Uses SvelteKit's built-in `dev` flag — no manual configuration needed
- Dynamic `import()` ensures MSW code is **tree-shaken from production builds**
- `onUnhandledRequest: 'bypass'` silences warnings for non-mocked requests (WebSocket, etc.)

### Running without a backend

```bash

npm install
npm run dev
```

### Adding new mock handlers

1. Add mock data to `src/lib/mocks/mock-data.ts`
2. Add the handler to `src/lib/mocks/handlers.ts` following the existing pattern:

```ts
import { http, HttpResponse } from 'msw';

http.get('/api/v1/your-endpoint', async () => {
  return HttpResponse.json({ /* your mock data */ });
});
```

3. For streaming endpoints (SSE), use `ReadableStream` + `HttpResponse` — see the chat completions handler for reference.

## The Shared Contract

The **single source of truth** is `api-spec/openapi.json` at the project root. This spec is auto-generated by FastAPI from the backend's route definitions and Pydantic models.

**For all teams:**
- Read the spec to know exact endpoints, request/response shapes
- If the backend changes, the spec changes — frontend CI catches mismatches
- Contract testing (e.g., Schemathesis) validates the backend matches the spec

## Setting Up MSW in Other Services

### For Admin Dashboard (React + Vite)

1. Install MSW:
   ```bash
   cd services/admin-dashboard
   npm install msw --save-dev
   npx msw init public/ --save
   ```

2. Create mock handlers in `src/mocks/` following the OptamusUI pattern

3. Wire into the app in `src/main.tsx`:
   ```tsx
   if (import.meta.env.DEV) {
     const { worker } = await import('./mocks/browser');
     await worker.start({ onUnhandledRequest: 'bypass' });
   }
   ```

4. Generate TypeScript types from the OpenAPI spec:
   ```bash
   npx openapi-typescript ../../api-spec/openapi.json -o src/types/api.ts
   ```

### For MainBackend (Python/FastAPI)

The backend already generates the OpenAPI spec automatically. To run the backend in stub mode:

1. Define Pydantic models and route decorators
2. Return placeholder responses:
   ```python
   @router.get("/endpoint")
   async def get_endpoint():
       return {"status": "stub", "data": []}
   ```
3. Hit `/openapi.json` to get the updated spec

## Switching from Mock to Real Backend

**No code changes needed.** The app code is identical — only the network layer changes:

1. Start the real backend (e.g., `docker compose up main-backend`)
2. Stop MSW by either:
   - Running without `npm run dev` (use production build)
   - Setting `VITE_USE_MOCKS=false` in `.env`
3. The Vite proxy in `vite.config.ts` forwards requests to the backend:
   ```ts
   server: {
     proxy: {
       '/api': 'http://127.0.0.1:9000',
       '/ws': { target: 'http://127.0.0.1:9000', ws: true },
     }
   }
   ```

## Best Practices

- **Mock at the network layer, not the component layer** — MSW intercepts real `fetch` calls, so your app code never knows it's talking to a mock
- **Use realistic data** — `mock-data.ts` uses real model IDs, proper timestamps, and realistic shapes to surface UI bugs early
- **Model streaming from day one** — the chat completions mock uses SSE streaming, so the state layer is designed for streaming from the start
- **Keep mocks in sync with the spec** — when the OpenAPI spec changes, update handlers and mock data to match
- **Test with mocks, verify with real backend** — use MSW handlers in integration tests, then run the same tests against the real backend in CI

## Backend Architecture (for Frontend Developers)

### Service Responsibilities

```
User request → MainBackend → Bifrost → AI Provider
                   │
                   ├── Validates auth (JWT, API key, or Zitadel token)
                   ├── Checks rate limits & spending limits
                   ├── Logs usage for billing
                   └── Forwards to Bifrost
```

| Service | Role | What it does NOT do |
|---------|------|---------------------|
| **Zitadel** | Auth/IAM (OIDC) | User management, login, password — Zitadel is the source of truth |
| **Bifrost** | AI Gateway | Provider routing, model listing, API conversion — Bifrost handles all AI provider logic |
| **MainBackend** | Business Logic | Auth validation, rate limiting, usage tracking, API key management, chat history, RAG, tools |

### What MainBackend KEEPS (your endpoints won't change)

MainBackend retains all app-specific features that frontends depend on:

| Category | Endpoints | Notes |
|----------|-----------|-------|
| **Auth (backward compat)** | `/api/v1/auths/signin`, `/api/v1/auths/signout` | Still works for admin dashboard |
| **Users** | `/api/v1/users/*` | User CRUD, settings, permissions |
| **API Keys** | `/api/v1/keys/*` | Developer `op-` key management |
| **Usage** | `/api/v1/usage/*` | Usage analytics |
| **Billing** | `/api/v1/billing/*` | Plan listing |
| **Chats** | `/api/v1/chats/*` | Chat history |
| **Models** | `/api/v1/models/*` | Model listing (now proxied to Bifrost) |
| **Config** | `/api/v1/configs/*` | App settings |

### What Changed (thin proxy layer added)

New thin proxy endpoints that delegate to Bifrost:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/v1/chat/completions` | POST | Chat completions → Bifrost |
| `/v1/models` | GET | Model listing → Bifrost |
| `/v1/completions` | POST | Text completions → Bifrost |
| `/v1/embeddings` | POST | Embeddings → Bifrost |
| `/admin/models` | GET | Admin model listing → Bifrost |

These run alongside the existing `/openai/...` and `/ollama/...` paths for backward compatibility.

| Current Path | Replacement | Timeline |

| `/api/v1/auths/signup` | Zitadel OIDC signup | When frontend teams update |
| `/api/v1/auths/update/password` | Zitadel password management | When frontend teams update |
| `/openai/...` | `/v1/...` (via Bifrost proxy) | Already works in parallel |
| `/ollama/...` | `/v1/...` (via Bifrost proxy) | Already works in parallel |
